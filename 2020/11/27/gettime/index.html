获取当前时间的性能坑

2020-11-27
performance

一般获取当前时间戳是使用

// 对于c++
auto start = std::chrono::high_resolution_clock::now();
// auto end = std::chrono::high_resolution_clock::now();
// double tot = std::chrono::duration_cast<std::chrono::duration<double>>(end - start).count();

// 对于linux/c
#include <sys/time.h>
timeval tv;
gettimeofday(&start, NULL);

在x86架构下，这俩函数都是用了vdso(virtual dynamic shared object)/vsyscall来避免syscall的使用从而加速，但这里有个坑是如果编译时采用-static的话，最后得到的二进制文件实际是包含syscall的，这极大降低了它的性能

我理解是-static所链接的静态库里没有实现对于时间获取的用户态加速，以后遇到这个问题保险做法还是得objdump检查下具体实现….

下面是一些性能数据

  • 动态链接

    ns/opevaluating get_time
    43.97gettimeofday
    47.80std::chrono::high_resolution_clock::now

  • 静态链接(使用-static)

    ns/opevaluating get_time
    527.65gettimeofday
    526.82std::chrono::high_resolution_clock::now

可以看到有超过10倍的性能差距….
